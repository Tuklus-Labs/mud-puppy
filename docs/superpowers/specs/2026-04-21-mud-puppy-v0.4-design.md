# mud-puppy v0.4 — Stream, Sprint, Show

Date: 2026-04-21
Status: Approved
Scope: RAM→GPU layer streaming, training throughput wins, standalone Phos studio UI

## Goals

1. True per-layer RAM→GPU streaming so a 70B base can train on a 24GB VRAM card backed by 192GB system RAM, without OOM and with acceptable throughput.
2. Measurable training speedups on the 7900 XTX via allocator config, sequence packing, and a unified offload path.
3. Replace the current HTML dashboard and Rich TUI with a single standalone Phos desktop app — `mud-puppy-studio` — that owns the entire fine-tuning UX (configure, launch, monitor, compare, manage).

Out of scope: multi-GPU/distributed; JAX backend changes; new quantization formats; Windows/macOS; remote attach; partitioned optimizer (stays a stub).

## Context

### Current state

- `StreamWrapper` in `mud_puppy/trainer.py` moves every transformer layer to GPU before forward and keeps them resident through backward; it only offloads after optimizer.step(). This saves VRAM between steps but every layer is resident during forward/backward.
- Two parallel offload implementations exist: `ZeroOffloadCallback` in `trainer.py` (simple sync copy) and `CPUOffloadOptimizer` in `zero_offload.py` (more featureful, pinned, still synchronous). Maintaining both is a bug farm.
- `DynamicBatchSampler` groups by token budget but still pads. For varied-length chat datasets the GPU wastes 30–60% of compute on pad tokens.
- `dashboard.html` is a 1184-line single-file Tempest Vector-Neon HTML/JS dashboard. `tui.py` is a Rich TUI alternative. Both are passive monitors — user launches training from a terminal, UIs attach.

### What ships on 7900 XTX

ROCm 7.1, PyTorch 2.10, Triton 3.6, 192GB DDR5, PCIe 4.0 x16. Single GPU. No distributed concerns.

## Section A — True RAM→GPU layer streaming

### Design

New module `mud_puppy/stream.py` replacing the existing `StreamWrapper`.

- Pin a host copy of every transformer block in pinned system RAM at load time.
- Two CUDA streams: `compute_stream` for forward pass, `h2d_stream` for layer transfer.
- Ring buffer of K=3 GPU slots (configurable via `prefetch_layers`): current layer, prefetch target, release slot.
- Pre-forward hook on layer N issues async H2D of layer N+1 on `h2d_stream` and records an event. Compute stream `wait_event`s that before executing N+1.
- Post-backward hook frees GPU slot of layer N-1 back into the ring and schedules the next prefetch.
- Embedding, lm_head, final layernorm stay permanently GPU-resident (small, always needed).
- LoRA adapters, when attached, stay permanently GPU-resident too. This eliminates the current "streaming + LoRA disabled" warning in `run_training`.

### Bandwidth sanity check

A 7B bf16 transformer block is ~200 MB. PCIe 4.0 x16 effective ≈ 28 GB/s → 7 ms per layer. TinyLlama layer forward on 7900 XTX ≈ 10–15 ms; 70B layer (~1.5 GB) transfer ≈ 55 ms which dominates but stays feasible with K=3 prefetch depth and micro-batching. Target: streaming overhead ≤ 20% vs. full-resident on 7B; ≤ 2x on 70B (where without streaming you can't run at all).

### Backward pass

Backward needs the same layer weights as forward. We keep weights in the ring buffer until their backward gradient has been produced. That means peak residency = (compute slot) + (H2D inflight slot) + (backward-held slots). For small K, use gradient checkpointing to avoid needing activations for all K slots at once.

### API

```python
from mud_puppy.stream import LayerStreamer

streamer = LayerStreamer(model, prefetch_layers=2)
# model's transformer blocks are now wrapped; forward/backward work unchanged
```

`run_training` picks this up when `config.stream=True`. If LoRA is active, the streamer pins adapter tensors before wrapping.

### Tests

- `tests/test_stream_prefetch.py` — mock 4-layer toy transformer, assert layers are NOT all resident at any single point (only K at a time), assert `h2d_stream` and `compute_stream` do overlap via recorded event timings.
- Integration: TinyLlama LoRA run with `--stream` completes without OOM on a 4 GB-limited torch memory pool (via `torch.cuda.set_per_process_memory_fraction`).

## Section B — Training speed wins

All ROCm-first, no new heavyweight deps.

### B1. Allocator

Change the default `PYTORCH_HIP_ALLOC_CONF` from `max_split_size_mb:128` to `expandable_segments:True`. Measured on Aegis' own workloads to reduce fragmentation dramatically at long-context training. `os.environ.setdefault` so users can still override.

### B2. Sequence packing

New module `mud_puppy/packing.py`. Replaces the "pad every batch to the longest sequence" behavior with concatenation of short sequences into a single packed row, using a block-diagonal attention mask and per-segment position IDs. For chat datasets with length variance from 50 to 2000 tokens, this commonly yields 1.5–3x throughput.

- New config flag: `pack_sequences: bool = False` (opt-in for v0.4, default-on in v0.5 after shakedown).
- Implementation: `PackedCollator(max_seq_length, pad_to_multiple_of=8)` — greedily bin-packs examples into rows of `max_seq_length`, emits `attention_mask` as a 2-D block-diagonal tensor AND `position_ids` as per-segment `[0..len-1]` ranges. Models that don't accept 2-D attention masks fall back to padded batching (log a warning, disable packing for the run).
- `DynamicBatchSampler` is kept for non-packing runs.

### B3. Offload consolidation

- Delete `ZeroOffloadCallback` from `trainer.py`. Sole offload path is `CPUOffloadOptimizer` in `zero_offload.py`.
- Switch `CPUOffloadOptimizer` internal AdamW to `torch.optim._multi_tensor.AdamW` (or the public foreach path on PyTorch 2.10+): ~2x CPU step time from foreach fusion on long parameter lists.
- Add async grad transfer: on `loss.backward()` completion, kick off D2H copy of the gradient to the already-pinned CPU accumulator on `h2d_stream` while the next microbatch's forward runs on `compute_stream`. Sync before optimizer step.
- Deprecate `PartitionedOptimizer` (keep class stub, emit DeprecationWarning, remove in v0.5). Streaming renders it obsolete for the single-GPU target.

### B4. Compile default

When `--compile` is passed, use `mode="reduce-overhead"` as default. Document the Triton-cache warm-up on first step. Skip compile entirely when `stream=True` (torch.compile does not yet play well with custom forward-pre-hooks at the block level).

### B5. Dataloader knobs

- `persistent_workers=True` and `prefetch_factor=4` when `dataloader_num_workers > 0`.
- Keep `pin_memory=True` (already on).

### Tests

- `tests/test_sequence_packing.py` — correctness: cross-segment attention is masked to zero, position IDs reset per segment, labels preserved with `-100` across segment boundaries.
- `bench/throughput.py` — TinyLlama + Alpaca-style mixed-length JSONL; report tokens/sec before vs. after (targets: ≥1.5x from packing alone on varied-length; streaming overhead ≤20% on 7B).

## Section C — `mud-puppy-studio` (Phos app)

Standalone desktop app. Not a browser dashboard. Not a TUI. Owns the whole fine-tuning UX.

### Architecture

```
mud-puppy-studio (Phos binary, ~1.5 MB, links libphos)
├── C++17 shell: GTK4 window + WebKitGTK webview
│   ├── TrainingManager  — spawn/kill Python training sidecar(s)
│   ├── WsBridge         — WebSocket client to sidecar :5980, forwards to webview as IPC events
│   ├── HfClient         — libcurl wrapper around api.huggingface.co
│   ├── DatasetPeek      — read first N lines of JSONL, detect format
│   └── CheckpointScan   — enumerate checkpoints / LoRA library dirs
│
└── Web layer (React + Vite, built to a static bundle embedded via phos asset-embed)
    ├── Launch pane      — model picker, dataset picker, method, hyperparams, live VRAM estimate
    ├── Monitor pane     — charts, streaming viz, GPU/vram, throughput, LoRA heatmap
    ├── Runs pane        — history, compare, re-run
    ├── Library pane     — checkpoints + LoRA adapters, merge/export
    └── Logs pane        — stderr tail from sidecar
```

Training sidecar is the existing `mud_puppy.cli` process invoked with `--monitor --monitor-port N`. The C++ `TrainingManager` spawns it, captures stderr, and opens a WS to `localhost:N`. JSON frames from the WS become IPC events in the webview — the same event names (`metrics`, `gpu`, `stream_stats`, `log_line`) get used across both sides.

### IPC surface (from `manifest.toml`)

```toml
[[ipc]]
name = "run.start"
args = { config = "TrainingConfig" }
returns = "RunHandle"

[[ipc]]
name = "run.stop"
args = { run_id = "string" }

[[ipc]]
name = "run.list"
returns = "RunSummary[]"

[[ipc]]
name = "hf.search_models"
args = { query = "string" }
returns = "HFModel[]"

[[ipc]]
name = "checkpoint.list"
args = { output_dir = "string" }
returns = "Checkpoint[]"

[[ipc]]
name = "dataset.preview"
args = { path = "string", n = "int" }
returns = "{ format: string, rows: any[] }"

[[event]] name = "metrics"
[[event]] name = "gpu"
[[event]] name = "stream_stats"
[[event]] name = "log_line"
[[event]] name = "run_complete"
```

TypeScript `.d.ts` generated via `phos types` — single source of truth.

### Protocol discipline

Training-process `stdout` is already a JSON stream for the monitor in theory, but to avoid the Forge stdout-vs-stderr trap (see `feedback_forge_sidecar_stdout.md`), WebSocket is the only JSON channel and stdout+stderr are captured as plain text and forwarded as `log_line` events for the Logs pane.

Motifs

- Inter-panel "vector wires" — thin animated polylines that thicken briefly on matching events.
- WebGL background shader — slow phase-grid vector field with subtle parallax on mouse. 30fps cap. Disable with `?chrome=minimal` or `Ctrl+Shift+M`.
- CRT scanlines kept but dimmed to 0.04 opacity (down from 0.08).
- Outer-glow only on active state or alert — no always-on bloom.

Key views
1. Loss chart — SVG line, 1k-point decimated ring, log-y toggle, step/wall-clock x toggle, hover crosshair, compare-two-runs overlay.
2. Throughput strip — tokens/sec, steps/sec, H2D bandwidth as large tabular digits over sparklines.
3. VRAM stack bar — model / optimizer / activations / kv / free / fragmentation, hover for `torch.cuda.memory_stats()` detail.
4. Streaming viz — vector stack of all transformer layers; GPU-resident layers glow cyan; prefetching layer pulses; CPU-resident layers dim.
5. LoRA heatmap — layer × rank, cell color by L2 norm, updated on the 50-step cadence.
6. Run list — left rail, 280px, active run with live status dot on top; past runs below.

### Tech choices

- React + Vite, single main chunk (no `manualChunks` — WebKitGTK TDZ lesson from Forge).
- Tailwind for layout + custom tokens file for the neon palette.
- D3 for SVG charts; twgl.js for the background shader.
- Root ErrorBoundary (Forge lesson — render error shouldn't wipe the app).
- Zustand for state.
- No Electron. No Node sidecar. C++ does filesystem, HTTP, process spawn directly.

## Test plan

1. `pytest tests/` — all existing tests pass.
2. New streaming and packing unit tests pass.
3. `bench/throughput.py` on TinyLlama + mixed-length JSONL:
   - Packing enabled vs. disabled: ≥1.5x tokens/sec.
   - Streaming vs. non-streaming on 7B bf16: streaming overhead ≤20%.
4. Manual: launch `mud-puppy-studio`, configure a TinyLlama LoRA run on a 100-row JSONL, hit Start, confirm all panels update, confirm logs stream, confirm the run finishes and appears in Runs pane.
5. Manual: kill the studio mid-run — ensure the sidecar training process also terminates (no orphaned Python).

## Risks and mitigations

- **ROCm + `torch.cuda.Stream` edge cases.** Validate event semantics with a 4-layer toy before wiring into the real training loop.
- **Gradient checkpointing × layer streaming** — activations and weights have different residency windows. Test with LoRA + `--stream` + `--use-gradient-checkpointing` specifically.
- **Packing correctness** — the block-diagonal mask must zero cross-segment attention exactly. Unit test with hand-computed attention scores.
- **WebKitGTK chunking** — single Vite chunk enforced via config, no manualChunks. Verified by Forge experience.
- **Sidecar zombie** — `TrainingManager` installs a `SIGCHLD` handler and on studio shutdown sends `SIGTERM` then `SIGKILL` after 10s grace (not 30 — mud-puppy has no multi-second cleanup work).
